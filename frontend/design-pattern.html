Design pattern before implementation

Warm light-mode commerce, built for asynchronous checkout.

A compact UI direction for Minishop: Anthropic-like calm surfaces, TWFoundry-style tokens, bilingual copy, and clear separation between request received, inventory reserved, and order confirmed.

Design decisions

Rules that keep the UI honest under load.

These decisions are intentionally concrete so the first Next.js implementation has fewer open questions.

Inventory display No optimistic decrement Only update from the SKU inventory projection endpoint.
Copy model en / zh-TW Use short state labels; keep event-sourcing terms out of primary buyer copy.
Layout Product first Show the purchase experience immediately, not a marketing landing page.
Spinning UI Active async only Use per-product spinners for reservation or projection work; stop when a projection-backed outcome arrives.

Responsive breakpoints

Device size changes product behavior.

Breakpoints define how product media, purchase controls, cart progress, notifications, and operator data reflow.

Phone 0-639px Single column. Product media first, purchase controls immediately below, checkout status and notifications stacked. Operator strip hidden by default.
Tablet 640-1023px Single or compact two-column layout. Keep Buy controls visible before long status history. Cart product rows stay full width.
Desktop 1024-1439px Two columns. Product media on the left, purchase and checkout panels on the right. Notification center stays near status.
Wide desktop 1440px+ Constrain content width. Do not stretch text lines or product panels endlessly. Operator strip may sit below status or in a secondary rail.

Breakpoints may change layout and density, but font size must not scale directly with viewport width.

Components

Small primitives before a UI library.

Start with local button, badge, panel, status row, language switcher, and operator metric components. Add headless primitives later only when behavior requires them.

Do

  • Use stable dimensions for buttons, counters, and status rows.
  • Separate request state, checkout projection state, and inventory projection state.
  • Use semantic status color only where state meaning changes.

Avoid

  • Do not show "reserved" copy after a successful POST alone.
  • Do not hide projection lag during benchmark work.
  • Do not use heavy gradients, nested cards, or decorative blobs.
